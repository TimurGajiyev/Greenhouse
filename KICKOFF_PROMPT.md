## Режим работы

MODE: BUILD

- **BUILD** — ты реализуешь код сам: research → план → код → тесты → отчёт. Код с построчными русскими комментариями. После каждого шага — отчёт с объяснением всех новых терминов, чтобы Тимур учился на твоей работе.
- **MENTOR** — ты даёшь только архитектуру, спецификации полей, формулы, acceptance criteria и заготовки-фрагменты; реализацию пишет Тимур сам, ты проверяешь его код и объясняешь ошибки.

(Чтобы сменить режим, Тимур меняет одну строку `MODE:` выше и говорит об этом.)

## Миссия

Довести GreenHouse от текущего состояния (шаги 1–3 из 9 закрыты, версия v0.2) до работающего ядра калькулятора: солнце → симулятор → экономика → оптимизатор-диспетчер → оптимизатор-сайзер → sensitivity. ### Шаг 4 — солнечный профиль (pvlib)

Цель: `src/solar.py` — почасовой годовой ряд выработки на 1 kWp установленной мощности (удельная выработка) для координат площадки из сценария.

Требования:
- Погодный год TMY (Typical Meteorological Year) через `pvlib.iotools`: основной путь — PVGIS (`get_pvgis_tmy`), fallback — NASA POWER. Объясни Тимуру, что такое TMY и компоненты облучённости GHI/DNI/DHI.
- Модель PVWatts из pvlib (упрощённая, но отраслевой стандарт для первого приближения); наклон/азимут — из сценария или разумный дефолт с пометкой `# ASSUMPTION`.
- Потери — как именованные параметры, даже если значения v1 простые: system losses по умолчанию pvlib + место под soiling (песок, ≈5%) и температурный derate.
- Индекс результата: tz `Asia/Aden`, длина и Δt согласованы с рядом нагрузки; API-стиль как в `profiles.py` (одна публичная функция + `_helpers`).
- Оффлайн-режим: если интернета нет — кэшированный погодный файл в `tests/data/` и честное сообщение об этом; никакой имитации данных.

Тесты: ночью выработка 0; значения в коридоре [0, ~1.2] kW/kWp; годовая удельная выработка в правдоподобном диапазоне для Йемена (~1500–2300 kWh/kWp — зафиксируй фактическое значение как regression-тест); длина и Δt согласованы с load-рядом; Δt берётся только из `timestep_hours`.

Done: pytest зелёный; в отчёте — типовые сутки (день/ночь) и объяснение каждого нового термина.

### Шаг 5 — симулятор (rule-based dispatch)

Цель: `src/simulate.py` — почасовой энергобаланс года при ЗАДАННЫХ размерах оборудования.

Правило приоритета (простая FSM): PV → нагрузка; избыток → заряд BESS (КПД η_charge, лимиты мощности PCS и ёмкости); остаток избытка → curtailment; дефицит → разряд BESS (η_discharge, пол soc_min); остаток дефицита → DG (в пределах размера); остальное → shortfall.

Каждый временной шаг формирует typed record (frozen dataclass или pydantic-модель): run_id, timestamp, все потоки kW, soc_kwh, source_model="rule_v1" — плюс assert энергобаланса inflow == outflow ± 1e-6. Δt — только из `timestep_hours`. Выход: DataFrame → Parquet в `results/` + manifest (JSON: run_id, git commit, hash входов).

Опциональность обязана работать: PV=None → нулевой ряд солнца, Battery=None → нет заряда/разряда, «дизель-онли» сценарий прогоняется без спец-веток.

Тесты: баланс на каждом шаге; SOC всегда в [soc_min·E, E]; «дизель-онли» работает; вендорский Йемен: shortfall == 0 — а если нет, это результат для доклада, не баг, который надо замаскировать.

Done: pytest; отчёт с 2–3 контрольными сутками (солнечный день / ночь / «плохой» день).

### Шаг 6 — KPI и экономика

Цель: `src/kpi.py` + `src/economics.py`.

KPI из симуляции: renewable fraction, kWh и $ дизельной генерации (через $0.26/kWh; литры — только при наличии удельного расхода, иначе `# ASSUMPTION`), curtailment %, LPSP = Σ shortfall / Σ load.

Экономика: CRF (capital recovery factor — объясни формулу и зачем он «размазывает» CAPEX по годам); annualized cost = CRF·CAPEX + O&M + fuel; NPC (Net Present Cost); LCOE = стоимость / полезная энергия; простая окупаемость против базовой линии «100% дизель @ $0.26/kWh».

Тесты: CRF против ручного расчёта; LCOE на вырожденном кейсе (одна технология, простые числа); согласованность единиц (Δt!).

Done: таблица KPI для `yemen_vendor` в отчёте, с интерпретацией на человеческом языке.

### Шаг 7 — оптимизатор-диспетчер (PuLP + HiGHS)

Цель: `src/optimize.py`, режим dispatch — размеры зафиксированы, солвер решает потоки (charge / discharge / dg / curtail / shortfall) на каждом t, минимизируя топливо + VOLL·shortfall (VOLL = value of lost load, $/kWh недопоставки — объясни).

Ограничения: энергобаланс; SOC-динамика stored[t] = stored[t−1] + Δt·(η·charge − discharge/η); лимиты мощностей; всё умножается на Δt.

Сверка здравого смысла: издержки солвера ≤ издержек rule-based симулятора на том же сценарии (объясни Тимуру, почему оптимизатор не может быть хуже жёсткого правила). Manifest дополняется: objective value, статус солвера, время решения.

Тесты: игрушечный 24-часовой сценарий с известным вручную оптимумом; `status == Optimal`.

Done: сравнение rule vs LP на Йемене в отчёте.

### Шаг 8 — оптимизатор-сайзер (главный ответ калькулятора)

Цель: режим sizing — `pv_kwp`, `batt_kwh`, `batt_kw`, `dg_kw` становятся непрерывными переменными решения ≥ 0.

Цель-функция: min CRF·CAPEX(размеры) + O&M + fuel + VOLL·Σ(shortfall·Δt).

Надёжность — переключаемая политика (объясни различия и покажи, что REopt делает так же — `min_load_met_annual_fraction`, `dvUnservedLoad`):
(a) hard: Σ shortfall == 0;
(b) LPSP ≤ x%;
(c) чистый VOLL в целевой функции.

Ограничения: как в шаге 7, плюс связь с размерами: soc ≤ batt_kwh; |charge|, |discharge| ≤ batt_kw; dg[t] ≤ dg_kw; pv_gen[t] = pv_kwp·profile[t]; площадь: pv_kwp · m²_per_kwp ≤ 8500 (значение m²_per_kwp — `# ASSUMPTION` с указанием источника).

simulate-режим тем же кодом: фиксация min = max (приём REopt). Постобработка: штуки = ceil(size / unit) по unit-полям схемы; вывод и в kW/kWh, и в штуках.

Тесты: игрушечный кейс с ручным оптимумом; фиксация размеров воспроизводит результат шага 7; сравнение с вендором (1500 / 3132 / 1000) — в отчёт, не в тест.

Done: отчёт «наш оптимум vs вендор» с интерпретацией: что означает разница, где вероятный over-spec, какие допущения могли её создать.

### Шаг 9 — sensitivity + Pareto + манифест

Цель: `src/sweep.py` — свипы поверх чистой функции ядра: цена дизеля ±50%; CAPEX BESS ±30%; стресс-сценарии (песчаная буря: PV ≈ 0 на N суток; топливный разрыв: DG = 0 на M суток); Pareto-фронт «стоимость vs надёжность» (свип цели LPSP) с поиском «колена» кривой.

Каждый прогон получает manifest; результаты — Parquet + сводная таблица.

Done: таблица/график Pareto + вывод «где колено и что это значит для закупки оборудования».
